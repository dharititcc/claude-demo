The platform

Every organization gets its own database.

A central database holds the tenant registry, users, plans, subscriptions, and platform audit trail. Everything else — customers, projects, tasks, files, roles — lives in a per-tenant database, provisioned automatically when an organization signs up.

That isolation is the platform's core promise. It is also its defining hazard.

The trap: a query hitting the wrong connection doesn't error — it silently reads another tenant's data, or dies far from its cause. Every model pins its connection on purpose.

The brief

The Super Admin could log in — but saw nothing.

A platform admin existed as a flag and a gate bypass, yet there was no organization list, no management screen, no way to act across tenants. The brief: build the full control plane — list, search, filter, dashboard stats, lifecycle actions, subscription detail, impersonation, and an audit trail.

The architecture challenge. Half the requirements are “across all organizations” — total users, projects, storage. But that data lives in thousands of separate databases. A naïve list would fan out one query per organization per page load: fine for three orgs, fatal for three thousand.

The requirement also assumed a server-rendered admin panel. The platform is an API consumed by a React SPA — so the panel became a React admin area over central-context endpoints, one stack, not two.

The build

Six phases. Backend first, so the UI met complete data.

Each phase shipped runnable and tested before the next began. The order was deliberate: make it correct and scalable, make it safe, then make it visible.

What the tests caught

Two latent bugs that would have destroyed data.

Neither was introduced by this work — both were dormant in the platform, and this module was the first code to exercise the paths that exposed them. Each was reproduced on a throwaway organization before any fix was trusted.

critical · data loss · lifecycle

A “soft delete” was dropping the tenant's database — irreversibly.

The tenancy library maps a model's deleted event to a job that drops the database. That event fires on a soft delete too — so a reversible-looking action was destroying an entire organization's data.

Fix

Database destruction is decoupled from the model lifecycle entirely. A soft delete keeps the database; only the deliberate, retention-gated purge command drops it. Verified live: soft delete → data survives; purge → dropped.

critical · silent · never worked

Restoring a deleted organization had never worked.

The tenant model funnels any undeclared attribute into a JSON blob on save. deleted_at was undeclared — so restore cleared it in memory but never wrote it back. A “restored” organization stayed invisible forever. Soft delete had only appeared to work because it writes via a direct query that bypasses the model.

Fix

Declared deleted_at a real column. Surfaced the moment a test actually restored an organization — exactly the kind of dormant bug a UI would have shipped over.

Security by design

“Log in as” — without becoming a liability.

Impersonation acts as the target user, so every permission check evaluates naturally — no special-casing. But the token is tagged and confined, and every guarantee below is proven by a test and was verified live against the running app.

✓

Confined to one organization

Blocked from the target's other orgs — 403, checked before membership.

✓

Cannot reach the admin surface

The impersonated user isn't a super admin — admin routes 404.

✓

Expires on its own

A short-lived token, ≤ 60 minutes, enforced by the auth layer.

✓

Revocable before expiry

“Stop” kills the token immediately — 401 on the next request.

✓

Never a super admin

You cannot impersonate another platform admin, yourself, or a non-member.

✓

Fully audited

Start and stop are both written to the central trail, with the real actor.

Verification

Real MySQL. Real browser. No shortcuts.

Every backend test provisions a real MySQL tenant database — SQLite cannot prove database-per-tenant isolation, which is the platform's central claim, so the suite pays the cost on purpose.

Every phase was then driven live in a browser against the running application: the dashboard's real rolled-up figures, the organizations table, an override taking a limit from 49 / 25 to 49 / 100, and impersonation start-to-stop.

What's not done — and why

The gaps, stated plainly.

A fabricated metric is worse than an absent one. These are the honest edges of what was built.

Needs Stripe

MRR, churn & dunning

Deliberately not built — they require live Stripe data this environment has never had. The scaffolding (subscription status, interval, plan) is in place for whoever wires real keys.

Needs Stripe

Billing paths unverified against live Stripe

Subscribe, swap, and invoice flows are written to Cashier's contract but have never touched the live API. Everything Stripe isn't responsible for is tested.

Environment

Test coverage never measured

No coverage driver exists in the local PHP build. CI is the first place a real percentage will appear — so no coverage number is claimed here.

Scope

Social login not built

Specified in the original brief, not yet implemented. Password, email verification, two-factor, and session management are.
